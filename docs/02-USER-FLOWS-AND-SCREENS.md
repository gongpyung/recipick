# 02. User Flows and Screens

## 핵심 유저 플로우
1. 사용자가 홈 화면에 진입
2. 유튜브 URL 입력
3. 추출 작업 시작
4. 진행 상태 확인
5. 결과 화면에서 재료/단계 확인
6. 인분 수 조절
7. 필요 시 결과 수정
8. 최근 결과로 다시 접근

---

## Screen A. 홈 화면
### 목적
최대한 빠르게 URL을 입력하게 만든다.

### 구성 요소
- 서비스 한 줄 설명
- URL 입력창
- `레시피 추출하기` 버튼
- 지원 형식 안내 (일반 영상 / 쇼츠)
- 최근 생성 레시피 1~5개

### 상태
- 기본 상태
- 잘못된 URL 입력 에러
- 최근 결과 없음 상태

### UX 원칙
- 홈은 단순해야 함
- CTA는 하나만 강하게 보여야 함

---

## Screen B. 추출 진행 화면
### 목적
사용자가 기다리는 동안 불안하지 않게 한다.

### 구성 요소
- 썸네일 / 영상 제목
- 단계형 진행 상태
  - URL 확인 중
  - 메타데이터 수집 중
  - 자막/설명 정리 중
  - 레시피 구조화 중
  - 결과 검증 중
- 실패 시 재시도 버튼

### 상태
- queued
- processing
- completed
- failed

### UX 원칙
- 10초 넘어가면 현재 단계가 보여야 함
- “멈춘 것 같음”이 느껴지면 안 됨

---

## Screen C. 레시피 결과 화면
### 목적
사용자가 실제로 요리할 때 참고 가능한 결과를 제공한다.

### 구성 요소
- 레시피 제목
- 원본 영상 링크 / 썸네일
- 기본 인분 수
- 인분 조절 컨트롤
- 재료 목록
- 조리 단계 목록
- 팁/메모
- 경고/불확실성 박스
- `수정하기` 버튼

### 모바일 레이아웃 권장 순서
1. 제목
2. 인분 조절 바
3. 재료 목록
4. 조리 단계
5. 팁
6. 경고

### 데스크톱 레이아웃 권장
- 왼쪽: 재료 목록
- 오른쪽: 조리 단계
- 상단: 제목 + 인분 조절

---

## Screen D. 편집 모드
### 목적
AI 결과를 사용자가 신뢰 가능한 개인 레시피로 다듬게 한다.

### 수정 가능 항목
- 제목
- 기본 인분 수
- 재료명
- 수량
- 단위
- 메모
- 조리 단계 문장
- 팁/메모

### UX 원칙
- 표보다 리스트/폼이 낫다
- 문서 편집기처럼 무겁게 만들지 않는다
- 수정 후 곧바로 반영된 결과를 보여준다

---

## Screen E. 최근 결과 / 히스토리
### 목적
방금 만든 결과를 다시 쉽게 찾게 한다.

### 구성 요소
- 최근 생성한 레시피 목록
- 제목
- 썸네일
- 생성 시간
- 마지막 수정 시간
- 다시 열기 버튼

### MVP 메모
- 로그인 없이 시작한다면 localStorage 기반 recent list로 충분
- 이후 DB 도입 시 확장

---

## Screen F. 실패 / 예외 화면
### 케이스
- 잘못된 URL
- 지원하지 않는 영상
- 자막/설명 부족으로 추출 불가
- 처리 중 내부 오류

### 사용자에게 보여줄 것
- 무엇이 실패했는지
- 다시 시도할 수 있는지
- 수동 입력 또는 다른 영상으로 넘어가야 하는지

---

## 주요 UX 결정
1. **인분 조절은 결과 화면 최상단**
2. **경고/불확실성은 숨기지 않고 명시**
3. **재료와 조리 단계를 섞지 않음**
4. **AI 결과는 기본적으로 수정 가능해야 함**
5. **결과 화면은 콘텐츠 소비형이 아니라 작업형 UI**
